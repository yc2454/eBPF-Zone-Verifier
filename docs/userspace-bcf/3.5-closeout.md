# Phase 3.5 — End-to-end go/no-go: CLEARED

**Date:** 2026-05-13
**Headline result:**

```
test_loader shift_constraint.bpf.o shift_constraint.bpf.o.bcf-bundle
  → SUCCESS: loaded prog fd=4
```

`shift_constraint.bpf.o` is the BCF benchmark program that the unpatched
kernel rejects with `invalid variable-offset read from stack`. With our
userspace-BCF bundle attached, the patched kernel runs `bcf_check_proof`
on the embedded proof, validates it against the bundle-stored goal,
hashes its own runtime refinement obligation, hits a matching bundle
entry by canonical hash, takes the discharge path, and accepts the
program — single-pass, no kernel↔userspace round-trip.

## Numbers

- **Bundle size on shift_constraint**: 32 900 → 11 924 bytes (−63%)
- **cvc5 proof size**: 32 656 → 11 688 bytes (−64%)
- Both reductions come from the kernel-shape DAG being substantially
  leaner (32-bit ALU ops where width fits, no redundant
  `EXTRACT(ZEXT(_))`, no const-offset embedding in pointer expressions).

## The five hard problems we solved

1. **Bundle wire format ABI mismatch.** The Rust writer packed entries at
   28 bytes, the kernel parser's natural struct layout was 32 bytes (u64
   `cond_hash` forced tail padding). Fix: `__attribute__((packed))` on
   the kernel struct.

2. **BCF infrastructure gate keyed on the wrong field.** The kernel's
   `env->bcf.available = is_priv && attr->bcf_buf_size` ignored bundles.
   Without `bcf_buf` allocated, refinement never triggered even with a
   valid bundle present. Fix: `|| attr->bcf_bundle_size`.

3. **Proof's claim vs bundle's stored goal mismatch.** zovia's
   `goal_root` was set to the raw refinement-condition (`oob`), but the
   proof's `assume` step claims `CONJ(path_cond, refine_cond)` — what
   cvc5 was actually asked to refute. Fix: `build_goal_root` mirrors
   kernel's `bcf_refine` CONJ-wrap.

4. **Kernel-vs-zovia BCF DAG divergence — the big one.** Even with
   matching bundle goal, the kernel's runtime CONJ structurally
   diverged: kernel used 32-bit AND for `w0 &= 255`, zovia widened to
   64-bit; kernel's refine_cond used `EXTRACT_32(off) sgt VAL_32(15)`,
   zovia used `off_64 sgt VAL_64(-1)`. Five sub-fixes:
   - Width-aware symbolic builders + `bcf_expr32` peeling
   - `RegBounds` + `reg_expr` caching with kernel's fit_u32/fit_s32 paths
   - ALU transfers thread width through to BCF ops
   - Branch path-cond uses `reg_expr(_, jmp32)`
   - `refine_stack` mirrors `__bcf_refine_access_bound` exactly,
     including K-folding into the predicate constant

5. **Kernel canonical_hash port bug.** Final blocker. `bcf_slot_map_build`
   iterated by record index with `exprs[i]`, but the kernel's flat-u32
   layout makes `exprs[i]` walk consecutive 4-byte slots, not consecutive
   expression headers. Pre-fix the kernel emitted 5 bytes of nonsense
   for any input. Fix: iterate by slot offset (`slot += 1+vlen`) and
   store identity in the slot_map. See
   `.claude/memory/feedback_kernel_canonical_hash_layout.md` for the
   full lesson (porting a struct-array algorithm across storage
   layouts is hazardous; dump byte streams to diagnose).

## What's still open (Phase 4 entry list)

- **Step 6 corpus regression check.** The width-aware ALU rewrites
  could in principle break programs in the Phase-2 corpus that
  previously coincidentally passed with the wrong DAG shape. Need to
  re-run the 24/42 cilium baseline and 7/9 collected/ to confirm no
  regressions. Mechanical at this point — `./scripts/sweep_new_only.sh`
  or equivalent.
- **Step 7 multi-program / multi-site validation.** shift_constraint
  is one program with one refinement site. The bundle protocol's full
  value shows up when multiple sites coexist in one program (e.g.,
  cilium programs with several variable-offset accesses).
- **bpftool `--bcf-bundle` flag** (B.5 follow-on). Currently `test_loader`
  is the only tool that loads a bundle; bpftool support is what makes
  this usable beyond the project's own test fixtures.
- **Phase 4 paper measurement.** Compare verification time / proof
  size / kernel LOC vs BCF's original suspend/resume on a real corpus.
- **Structural equiv check in `bcf_bundle_try_discharge`.** Currently
  hash match is sufficient. Per the discharge function's TODO comment
  (`kernel/bpf/bcf_bundle.c`), upstream-ready needs a
  `__expr_equiv(kernel_goal, bundle_goal)` confirmation after the hash
  hit. 64-bit hash collisions are vanishingly rare but soundness
  requires the explicit check.
